## 1. 问题描述

### 1.1 问题现象

用户报告解密功能失败，日志显示以下错误：

```
[ERROR] - ✗ 解密失败: 01 We've Got Love_hires.mflac - Invalid file size: 106611442
```

**受影响文件统计**：
- 文件 1：`01 We've Got Love_hires.mflac` - 106611442 字节（1.06 GB）- ❌ 失败
- 文件 2：`02-1.mflac` - 16700862 字节（15.9 MB）- ❌ 失败
- 文件 3：`02-2.mflac` - 16487814 字节（15.7 MB）- ❌ 失败
- 文件 4：`02-3.mflac` - 12248870 字节（11.7 MB）- ❌ 失败
- 文件 5：`02-4.mflac` - 10699578 字节（10.2 MB）- ❌ 失败

### 1.2 根本原因

**位置**：`hook_qq_music.js:57-61`

**问题代码**：
```javascript
if (fileSize == 0 || fileSize > 100 * 1024 * 1024) {
    console.log("[ERROR] Invalid file size: " + fileSize);
    EncAndDesMediaFileDestructor(EncAndDesMediaFileObject);
    return "Invalid file size: " + fileSize;
}
```

**原因分析**：
- 文件大小限制设置为 **100MB**
- 现代无损音频文件（尤其是 hires 音频）很容易超过这个限制
- 用户报告的最大文件为 **1.06GB**，远超 100MB 限制
- 错误信息只显示原始字节数，不直观（如 `106611442`）

### 1.3 影响范围

- **影响文件**：所有大于 100MB 的加密音频文件
- **影响功能**：批量解密功能
- **影响用户**：使用 hires 音质的用户

## 2. 修复方案

### 2.1 方案选择

经过评估，选择了**方案 3（最佳）：增大限制 + 详细错误信息**

**选择理由**：
1. ✅ **增大到 2GB**：适应现代高音质音频需求
2. ✅ **详细错误信息**：显示 MB 单位和文件名，提升用户体验
3. ✅ **保留安全验证**：仍然检查文件大小为 0 的情况
4. ✅ **代码清晰**：易于维护和调试

### 2.2 修改内容对比

#### 修改前（第 57-61 行）

```javascript
if (fileSize == 0 || fileSize > 100 * 1024 * 1024) {
    console.log("[ERROR] Invalid file size: " + fileSize);
    EncAndDesMediaFileDestructor(EncAndDesMediaFileObject);
    return "Invalid file size: " + fileSize;
}
```

#### 修改后（第 57-62 行）

```javascript
if (fileSize == 0 || fileSize > 2 * 1024 * 1024 * 1024) {
    var sizeInMB = (fileSize / (1024 * 1024)).toFixed(2);
    console.log("[ERROR] Invalid file size: " + srcFileName + " (" + sizeInMB + " MB)");
    EncAndDesMediaFileDestructor(EncAndDesMediaFileObject);
    return "Invalid file size: " + sizeInMB + " MB";
}
```

### 2.3 代码变更明细

| 项目 | 修改前 | 修改后 | 变更 |
|------|--------|--------|------|
| 文件大小限制 | 100 * 1024 * 1024 | 2 * 1024 * 1024 * 1024 | 增大 20 倍 |
| 限制值 | 104,857,600 字节（100MB） | 2,147,483,648 字节（2GB） | 增大 20 倍 |
| 错误信息 | 原始字节数 | MB 单位 + 文件名 | 更友好 |
| 临时变量 | 无 | `sizeInMB` | 新增 1 个 |
| 行数 | 5 行 | 6 行 | +1 行 |
| 代码复杂度 | 低 | 中等 | 略微增加 |

## 4. 风险评估

### 4.1 潜在风险

| 风险项 | 风险等级 | 描述 | 缓解措施 |
|--------|----------|------|----------|
| 2GB 限制仍不够用 | 🟡 中等 | 极少数超大型音频文件可能超过 2GB | 根据实际使用情况调整，如需要可进一步增大到 4GB |
| MB 转换精度问题 | 🟢 低 | `.toFixed(2)` 保留两位小数 | 精度足够，满足实际需求 |
| 大文件导致内存问题 | 🟢 低 | 2GB 文件分块读取，每次 64KB | 已有分块机制，内存占用稳定 |
| 向后兼容性 | 🟢 低 | 只是增大限制，不影响小文件 | 完全向后兼容，无破坏性变更 |

### 4.2 缓解措施

1. **监控实际使用情况**：
   - 观察解密日志，统计实际文件大小分布
   - 如果发现大量文件接近 2GB 限制，考虑进一步增大

2. **添加性能监控**：
   - 在解密日志中记录每个文件的解密时间
   - 识别异常慢的文件，可能需要优化

3. **保持备份**：
   - Git 管理版本历史，可以随时回滚
